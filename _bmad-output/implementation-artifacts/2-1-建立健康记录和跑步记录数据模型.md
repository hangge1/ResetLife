# Story 2.1: 建立健康记录和跑步记录数据模型

Status: done

## Story

As a 系统,
I want 保存健康记录和跑步记录的源事实数据,
so that 后续记录、历史和趋势都基于一致的数据。

## Acceptance Criteria

1. Given 用户需要保存健康和跑步记录, when 数据库 schema 扩展, then 系统创建健康记录表，且 `localDate` 唯一。
2. 系统创建跑步记录表，同一个 `localDate` 可以有多条跑步记录。
3. 记录使用字符串 UUID 主键。
4. 日期字段保存为 `YYYY-MM-DD`。
5. 体重、距离、围度、心率、步幅、步频、配速按架构约定保存为标准内部单位。
6. repository 提供创建、更新、查询、删除记录的基础方法。
7. UI、server action、route handler 不直接写数据库；读写必须经由 `features/records` 的 service/repository。

## Dev Notes

### Existing Patterns Reused

- 数据库访问模式沿用 `features/access/repositories/access-repository.ts`：repository 返回 `{ ok, data }` 或 `{ ok: false, error }`。
- 默认数据库连接、迁移和测试临时 SQLite 沿用 `db/client.ts`、`scripts/migrate-db.mjs`、现有测试中的 `migrate(db, { migrationsFolder: "./db/migrations" })`。
- 本故事未新增 UI 或 server action；后续 records actions 必须在入口复用 `requireTrustedDevice()`。

### Scope Boundaries

- 未实现记录录入 UI；Story 2.2/2.3 负责表单和 server actions。
- 未实现趋势、BMI、目标进度、鼓励文案；Epic 3 负责 read-model 计算。
- 未引入外部数据库或 ORM 替代方案。
- 未在 UI、route handler、server action 中散落 SQL。
