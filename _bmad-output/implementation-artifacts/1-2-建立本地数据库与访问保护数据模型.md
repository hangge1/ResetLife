# Story 1.2: 建立本地数据库与访问保护数据模型

Status: done

## Story

As a 系统,
I want 保存访问密码哈希和受信设备记录,
so that 个人健康数据可以被轻量保护.

## Acceptance Criteria

1. Given 应用需要保存访问保护状态, when 数据库 schema 初始化, then 系统创建访问密码元数据和受信设备相关表。
2. 访问密码只能以哈希形式保存；本故事不得保存明文访问密码。
3. 受信设备保存服务端可校验的设备标识记录；不得只依赖浏览器本地状态。
4. 数据访问通过 Drizzle repository，不在 UI、route handler、server action 或 scheduler 中散落 SQL。
5. 数据库使用本地 SQLite，并且数据文件路径可配置，默认落在可备份的本地 `data/` 目录。

## Dev Notes

### Scope

本故事只建立本地数据库、访问保护 schema、迁移和 repository。不要实现以下内容：

- 不做访问密码创建页面、验证页面、cookie/localStorage 写入或设备信任交互；这些属于 Story 1.3/1.4。
- 不移动现有首页、记录、历史、目标、设置页面到 `(protected)` 路由组；保护路由接入在后续故事完成。
- 不创建健康记录、跑步记录、目标、提醒、SMTP 或 settings 表；这些属于后续 Epic 2-4。
- 不引入 Clerk/Auth.js/OAuth/账号注册/多用户/角色权限。
- 不把密码、设备原始标识或 SMTP 密码写入环境变量、浏览器本地状态或明文数据库字段。

### Current Repository State

- Story 1.1 已完成并通过审查，当前是根目录 Next.js 16 App Router 项目。
- 已有目录：`app/`、`components/`、`components/ui/`、`components/layout/`、`features/access/components/`、`features/dashboard/components/`、`lib/`、`tests/`。
- 已有质量脚本：`npm test`、`npm run typecheck`、`npm run lint`、`npm run build`。
- `npm run typecheck` 已改为 `next typegen && tsc --noEmit`，不要回退。
- 现有测试 `tests/app-shell.test.mjs` 会启动或复用 Next dev server 做路由烟测；新增数据库测试应避免固定端口和长期占用进程。

### Architecture Compliance

必须遵守架构 Spine：

- `db/` 拥有 schema、migration 和数据库连接。
- `features/access/repositories/` 拥有访问保护持久化读写。
- UI、route handler、server action、scheduler 不得直接访问 SQLite 或写 SQL。
- MVP 使用单一本地 SQLite 数据库，应用代码只能通过 Drizzle repository 访问数据库。
- 访问保护是“设备信任”，不是账号系统：服务器保存访问密码哈希和受信设备记录；浏览器后续只持有随机设备标识，服务器保存并校验其哈希。
- 所有 ID 使用字符串 UUID；时间戳保存 ISO 字符串。
- 敏感值不得明文回显或明文保存；访问密码必须哈希保存，设备标识也只保存哈希。
- MVP 目标是长运行 Node.js 进程并拥有本地文件系统写权限；数据目录必须可持久化备份。

### Data Model Guidance

建议 schema 名称保持和架构一致：

```text
db/
  client.ts
  schema.ts
  migrations/
features/
  access/
    repositories/
      access-repository.ts
```

建议实体：

```text
accessSecrets
  id: text primary key UUID
  passwordHash: text not null
  passwordHashAlgorithm: text not null
  createdAtIso: text not null
  updatedAtIso: text not null

trustedDevices
  id: text primary key UUID
  deviceIdentifierHash: text not null unique
  displayName: text nullable
  userAgent: text nullable
  createdAtIso: text not null
  lastSeenAtIso: text not null
  revokedAtIso: text nullable
```

`accessSecrets` 是访问密码元数据表。MVP 只有一个访问密码，但不要用明文配置文件替代数据库表。可以通过 repository 保证只保留一条当前访问密码记录，具体 UI 流程在 Story 1.3 实现。

### Library / Framework Requirements

- 当前官方 npm registry 查询结果（2026-06-26）：`drizzle-orm` latest 为 `0.45.2`，`drizzle-kit` latest 为 `0.31.10`，`better-sqlite3` latest 为 `12.11.1`，`@types/better-sqlite3` latest 为 `7.6.13`。
- Drizzle 官方 SQLite 文档说明其原生支持 `better-sqlite3` 驱动，并使用 `drizzle-orm/better-sqlite3` 初始化 client。
- Drizzle Kit 官方流程支持 `drizzle-kit generate` 生成迁移、`drizzle-kit migrate` 应用迁移。本故事应使用迁移文件，不要只依赖开发期 `push`。
- 不要采用 libSQL/Turso、Prisma、Kysely、TypeORM 或手写全局 SQL helper；架构已锁定 Drizzle + better-sqlite3 + SQLite。

### Testing Requirements

新增数据库测试应做到：

- 使用临时目录和临时 SQLite 文件，不污染默认 `data/slimming-assistant.sqlite`。
- 能在 Windows PowerShell 和 CI 环境中运行。
- 不依赖真实浏览器 cookie/localStorage。
- 验证 repository 行为，而不是只做源码字符串匹配。
- 继续保留 Story 1.1 的应用 shell 路由烟测。

完成实现前必须运行：

```text
npm test
npm run typecheck
npm run lint
npm run build
```

### Anti-Patterns to Avoid

- 不要把 SQLite 连接、SQL 语句或 Drizzle 查询写进 `app/` 页面组件。
- 不要在 client component 中导入 `db/client.ts`、`db/schema.ts` 或 repository。
- 不要保存明文 `password`、`deviceId`、`deviceIdentifier`。
- 不要用 localStorage 作为唯一受信设备来源；本故事必须建立服务端可校验的数据库记录。
- 不要创建 UI 假数据来证明数据库已完成；用 repository 测试证明。
- 不要因为当前只有一个人使用就省略访问保护数据模型。
