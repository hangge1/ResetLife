# Story 1.4: 未受信浏览器访问验证与受信设备管理基础

Status: done

## Story

As a 已设置访问密码的个人用户,
I want 新浏览器需要输入访问密码，而受信浏览器可以直接进入,
so that 我的个人健康数据不会被未验证浏览器直接访问。

## Acceptance Criteria

1. Given 服务端本地已有访问密码, when 未受信浏览器打开应用, then 系统显示中文访问密码验证页面。
2. 输入错误密码时显示中文错误，不暴露底层错误。
3. 输入正确密码后浏览器生成设备标识并保存到 httpOnly cookie。
4. 服务端记录该浏览器为受信设备，数据库只保存设备 token hash。
5. 受信浏览器再次访问时跳过验证。
6. 未受信浏览器不能访问受保护页面或 server entrypoint。
7. 清除 cookie、更换浏览器或隐私模式访问时需要重新验证。

## Dev Notes

### Scope

本故事负责“已有访问密码后的未受信浏览器验证”和“受信设备跳过验证”。不实现设置页里的受信设备列表、移除设备、修改访问密码；这些属于 Story 4.7。不实现健康记录、跑步记录、目标、提醒、SMTP。

### Architecture Compliance

- 访问保护在服务端执行，不依赖客户端判断。
- 浏览器 cookie 保存原始设备 token；数据库只保存 `hashDeviceToken(token)`。
- 受保护页面复用 `requireTrustedDevice()`，避免每个页面各自拼装逻辑。
- 明文访问密码只在验证 action/service 中短暂存在，不写入数据库、日志、URL 或 cookie。
- 验证错误对用户只显示中文通用错误。
- 访问保护保持轻量设备信任模型，不新增账号、多用户、角色、OAuth 或第三方认证。
