# Story 1.3: 首次访问创建访问密码

Status: done

## Story

As a 首次访问的个人用户,
I want 第一次打开应用时创建访问密码,
so that 后续访问个人健康数据前有基本保护.

## Acceptance Criteria

1. Given 服务器本地还没有访问密码, when 用户打开应用, then 系统显示中文“创建访问密码”页面。
2. 用户可以输入并确认访问密码。
3. 密码不一致或不符合校验规则时显示中文字段错误。
4. 创建成功后服务端只保存密码哈希，不保存明文密码。
5. 创建成功后当前浏览器被标记为受信设备。
6. 创建成功后用户进入受保护的应用框架。

## Dev Notes

### Scope

本故事只处理“服务器本地还没有访问密码”的首次创建流程。

不要实现以下内容：

- 不做已有访问密码时的新浏览器输入密码验证；这是 Story 1.4。
- 不做受信设备列表、移除设备、修改访问密码；这些属于 Story 4.7。
- 不引入账号、注册、OAuth、Clerk/Auth.js、角色或多用户模型。
- 不创建健康记录、跑步记录、目标、提醒或设置业务能力。

### Existing Code To Reuse

- `db/client.ts`：默认连接会自动执行 Drizzle migrations，支持 `SQLITE_PATH`。
- `db/schema.ts`：已有 `accessSecrets` 和 `trustedDevices`。
- `features/access/repositories/access-repository.ts`：已有访问密码哈希和受信设备 repository。
- `components/ui/button.tsx`：按钮默认 `type="button"`，在表单 submit 按钮上必须显式 `type="submit"`。
- `app/page.tsx`：当前首页是 server component，可在进入页面前调用服务端检查并 `redirect("/access/create")`。
- `tests/app-shell.test.mjs`：现有路由烟测会访问 `/`、`/records`、`/history`、`/goals`、`/settings`；新增访问保护后必须保持测试可控。

### Architecture Compliance

- UI 页面和 server action 不得直接写 SQLite；写入必须通过 `features/access/services/` 再进入 `features/access/repositories/`。
- 明文密码只能在 action/service 内短暂作为输入存在，不能写入数据库、日志、URL、cookie、localStorage 或 story/debug 输出。
- 访问密码必须保存为哈希。建议格式：`scrypt:v1:<N>:<r>:<p>:<saltBase64>:<digestBase64>`。
- 设备 token 必须是高熵随机值；浏览器 cookie 保存原始 token，数据库只保存 token hash。
- cookie 名建议集中常量，例如 `slimming_device_token`，后续 Story 1.4 复用。
- 当前浏览器被信任时，repository 写入 `trustedDevices.deviceIdentifierHash`，并记录 user agent、创建时间、最近访问时间。
- 所有错误给 UI 的文本必须是中文；底层异常不得原样暴露到页面。

### UX Requirements

- 页面标题使用“创建访问密码”。
- 表单标签建议：`访问密码`、`确认访问密码`。
- 错误示例：`请输入访问密码`、`访问密码至少需要 8 个字符`、`两次输入的访问密码不一致`、`访问密码已经创建，请验证后访问`。
- 页面保持简洁、数据优先风格；不要 hero、营销文案、插画或英文主界面。
- 移动端输入控件高度至少 44px，错误文本靠近对应字段。

### Testing Guidance

- 测试不得依赖默认 `data/slimming-assistant.sqlite`；使用临时 `SQLITE_PATH`。
- 如果页面/路由烟测需要访问已初始化状态，可以在测试启动前用临时数据库写入访问密码哈希和受信设备，或提供仅测试使用的环境变量绕过首次创建检查。优先使用临时数据库，避免生产代码留下测试后门。
- 服务测试应验证数据库列和 repository 数据不包含明文密码或原始设备 token。
- 继续保留 Story 1.1/1.2 的现有测试。
