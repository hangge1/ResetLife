# 瘦身助手体验规范

## 基础

瘦身助手是个人单用户响应式网页应用，用于记录健康数据和跑步数据、查看目标进度、接收提醒。`DESIGN.md` 负责视觉语言；本文负责页面、行为、状态、流程和可访问性。

产品第一屏就是实际仪表盘，不做落地页。核心体验闭环是：设置目标 -> 记录今天 -> 查看进展 -> 收到提醒 -> 继续行动。

## 信息架构

| 页面 | 入口 | 目的 |
|---|---|---|
| 访问验证 | 未受信浏览器首次访问 | 创建或输入访问密码，避免暴露个人数据 |
| 首页 | 打开应用 / 导航 | 今日状态、健康和运动摘要、目标进度、预计达成时间、鼓励文案 |
| 今日记录 | 首页主按钮 / 导航 | 录入或更新今日健康记录，添加跑步记录 |
| 历史 | 导航 / 指标下钻 | 浏览、筛选、查看、编辑和删除健康记录与跑步记录 |
| 目标 | 导航 / 首页目标卡 | 设置健康目标和每周跑步目标 |
| 提醒 | 设置 / 首页提醒 | 配置提醒时间、站内提醒和邮件提醒 |
| 设置 | 导航 | 个人资料、SMTP、趋势阈值、访问密码、受信设备 |

桌面端使用左侧常驻导航。移动端使用顶部标题栏和底部导航：首页、记录、历史、目标、设置。

视觉布局参考：[关键页面视觉稿](mockups/key-screens.html)。规范与视觉稿冲突时，以 `DESIGN.md` 和本文为准。

## 文案语气

界面文案使用中文，语气平实、直接、具体。品牌气质见 `DESIGN.md`。

| 应该写 | 不要写 |
|---|---|
| 还差 2.4 公斤到目标体重。 | 快成功啦！燃起来！ |
| 数据不足，暂时无法估算达成时间。 | 预计很快达成。 |
| 今天还没有记录身体数据。 | 你又忘记了。 |
| 本周还差 2 次跑步。 | 任务失败。 |
| SMTP 测试邮件发送失败，检查主机、端口和安全模式。 | 邮件错误。 |

鼓励文案必须引用真实数据：完成记录、趋势改善、差距缩小或恢复连续记录。不要使用羞辱、医疗化判断或夸张确定性。

## 组件行为

| 组件 | 使用位置 | 行为规则 |
|---|---|---|
| 指标卡 | 首页 | 点击进入相关趋势详情或历史筛选。显示名称、数值、单位、变化说明和领域颜色。 |
| 目标进度卡 | 首页、目标 | 显示当前值、目标值、剩余差距、进度条和预计达成时间。无法估算时说明原因。 |
| 今日状态条 | 首页 | 展示健康记录、跑步记录和提醒状态。每一项都能跳转到对应操作。 |
| 记录表单 | 今日记录 | 健康记录每日一条，同日覆盖；跑步记录同日可添加多条。 |
| 图表卡 | 首页、历史 | 支持 7 天和 30 天视图。数据不足时显示空状态。 |
| 提醒条 | 首页 | 满足提醒条件时显示，不阻塞用户操作。 |
| 设置分组 | 设置 | 分组展示资料、SMTP、趋势阈值、访问保护，每组独立保存。 |
| 受信设备行 | 设置 | 显示浏览器/设备名称、创建时间、最近访问时间和移除操作。 |

## 状态模式

| 状态 | 页面 | 处理方式 |
|---|---|---|
| 首次访问 | 访问验证 | 创建访问密码，然后把当前浏览器标为受信设备。 |
| 未受信浏览器 | 访问验证 | 输入访问密码。验证成功后生成浏览器设备标识。 |
| 受信浏览器 | 全局 | 跳过访问验证，进入首页。 |
| 暂无数据 | 首页 | 显示设置清单：完善资料、设置目标、添加今日记录。 |
| 今日健康记录缺失 | 首页 | 显示提醒状态，并链接到今日记录。 |
| 今日没有跑步记录 | 首页 | 根据提醒设置显示中性或提醒状态。 |
| 数据不足无法估算 | 目标进度卡 | 显示“数据不足，至少需要 7 天跨度和 3 条有效记录。” |
| 趋势与目标相反 | 目标进度卡 | 显示“当前趋势暂不支持估算达成时间。” |
| SMTP 未配置 | 提醒 / 设置 | 邮件提醒标记为未完成，直到 SMTP 配置有效。 |
| SMTP 测试失败 | 设置 | 内联错误，提示检查主机、端口和安全模式。 |
| 记录保存成功 | 今日记录 | 显示内联成功提示，首页数据立即更新。 |
| 删除记录 | 历史 | 二次确认；删除后趋势重新计算。 |

## 交互规则

- 首页主操作是“记录今天”。
- 数值输入必须显示单位；错误信息显示在字段旁边。
- 日期默认今天；历史页面支持日期和周期筛选。
- 趋势卡使用 7 天 / 30 天分段控件。
- 设置页每个分组独立保存。
- SMTP 密码或授权码保存后不再明文显示。
- 删除记录、移除受信设备、清空 SMTP 配置都需要确认。

第一版禁止：拖拽排序、轮播仪表盘、移动端悬停操作、两层以上弹窗、打卡游戏化动画、核心历史记录无限滚动。

## 可访问性底线

- 响应式网页应用目标为 WCAG 2.2 AA。
- 所有输入都有可见标签和单位。
- 图表旁必须有文字摘要。
- 颜色不能作为唯一信息来源；健康和运动分类还要有文字标签。
- 所有可交互控件都有可见焦点状态。
- 移动端点击目标至少 44px 高。
- 错误信息要和字段关联，并能被读屏识别。
- 进度条要有可访问名称和数值。

## 响应式与平台

| 断点 | 行为 |
|---|---|
| `>= 1024px` | 左侧导航常驻。首页使用仪表盘栅格：状态、目标、趋势并列。 |
| `768px-1023px` | 导航可收起。首页尽量使用双列卡片。 |
| `< 768px` | 底部导航。卡片单列堆叠。表单单列。图表保留标签，不要求横向滚动。 |

网页应用应优先保证桌面端查看效率，同时跑步后用手机记录也必须顺畅。

## 关键流程

### 流程 1：跑步后记录

1. hangge 在受信浏览器打开瘦身助手。
2. 首页显示今日健康记录缺失、今日还没有跑步记录。
3. 他点击“记录今天”。
4. 今日记录页面打开，显示健康数据和跑步数据两组表单。
5. 他输入体重、腰围、腿围、体脂率、跑步时长、公里数、心率、步幅和步频。配速自动计算，也可以手动改。
6. 他保存记录。
7. **价值时刻：** 首页立即更新：今日记录完成、本周跑量进度增加、健康趋势使用最新记录、鼓励文案引用本次真实更新。

失败状态：数值字段非法时，字段旁显示错误，阻止保存，但保留已输入内容。

### 流程 2：查看目标进度

1. hangge 早上打开首页。
2. 他查看健康目标卡和运动目标卡。
3. 他把趋势卡从 7 天切换到 30 天。
4. 他从首页打开目标详情。
5. **价值时刻：** 详情页解释当前值、目标值、剩余差距，以及是否满足阈值可以估算达成时间。

失败状态：趋势与目标相反时，系统不估算达成时间，并说明原因。

### 流程 3：配置提醒和访问保护

1. hangge 打开设置。
2. 他填写 SMTP 主机、端口、安全模式、账号、授权码和发件人地址。
3. 他发送测试邮件。
4. 他配置提醒时间并开启邮件提醒。
5. 他查看受信设备，移除旧浏览器。
6. **价值时刻：** 设置页显示邮件提醒可用、受信设备已更新，未来新浏览器仍需输入访问密码。

失败状态：SMTP 测试失败时，设置页内联提示需要检查的配置项，提醒设置仍可继续编辑。

## 参考与反模式

- **参考个人仪表盘：** 第一屏直接提供状态和差距。
- **参考个人财务工具：** 目标进度显示当前、目标、剩余和趋势预测。
- **拒绝健身社交流：** 不做社交比较、排行榜或公开动态。
- **拒绝打卡游戏压力：** 不使用羞辱文案、烟花动画、连续打卡压迫感。
- **拒绝表格克隆：** 历史页面可扫描，但首页不是原始数据表。
