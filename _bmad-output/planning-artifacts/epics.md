# SlimmingAssistant - Epic Breakdown

## Overview

This document provides the complete epic and story breakdown for SlimmingAssistant, decomposing the requirements from the PRD, UX Design, SPEC, and Architecture requirements into implementable stories.

## Requirements Inventory

### Functional Requirements

FR1: 用户可以创建或覆盖当天健康记录，字段包括体重、腰围、腿围、体脂率；同一天再次提交时更新原记录，不新增第二条当天健康记录。

FR2: 用户可以创建跑步记录，字段包括日期、公里数、运动时长、配速、平均心率、平均步幅、步频；同一天允许多条跑步记录，配速可由系统计算并允许手动覆盖。

FR3: 用户可以查看、编辑和删除历史健康记录与跑步记录；编辑或删除后，相关趋势和目标进度必须重新计算。

FR4: 用户可以设置和更新健康目标，包括目标体重，以及可选的目标腰围、目标腿围、目标体脂率。

FR5: 用户可以设置和更新跑步目标，包括每周跑步次数和每周跑量。

FR6: 系统可以显示目标状态，包括未设置、进行中、已达成、落后；未设置目标时提示用户设置目标，未达到时展示距离目标的差距。

FR7: 首页可以展示今日状态，包括今日是否已完成健康记录、今日是否已有跑步记录、当前提醒状态。

FR8: 系统可以展示健康趋势，包括体重、腰围、腿围、体脂率、BMI 在最近 7 天和 30 天的变化；数据不足时显示空状态或数据不足提示。

FR9: 系统可以展示跑步趋势，包括最近 7 天和 30 天的跑量、跑步次数、配速变化或平均配速。

FR10: 系统可以展示健康目标和跑步目标的当前值、目标值、差距，并在达到配置阈值时使用简单线性趋势估算预计达成时间；数据不足或趋势反向时必须说明无法可靠估算。

FR11: 系统可以根据近期趋势、连续记录情况和目标进度展示简短中文鼓励文案；文案必须与当前数据状态一致，不使用羞辱性或医疗化表达。

FR12: 用户可以配置提醒规则，包括每日提醒时间、站内提醒开关、邮件提醒开关。

FR13: 当用户在提醒时间前未完成指定动作时，系统可以生成站内提醒；今日状态已完成时不得生成重复提醒。

FR14: 当用户启用邮件提醒且符合提醒条件时，系统可以使用配置界面中的 SMTP 参数发送邮件提醒，并保留邮件发送失败状态供用户查看。

FR15: 用户可以维护个人资料，包括昵称、身高、邮件提醒收件邮箱；设置身高后首页和健康趋势可以显示 BMI。

FR16: 用户可以在配置界面维护 SMTP 邮件发送参数，包括 SMTP 主机、端口、账号、密码或授权码、发件人地址、安全模式 None/SSL/STARTTLS，并可发送测试邮件。

FR17: 用户可以配置预计达成时间所需的最低数据条件，包括最低统计天数和最低有效记录数；系统下限分别为 7 天和 3 条，低于下限时阻止保存并显示说明。

FR18: 系统可以通过访问密码和受信浏览器设备标识保护个人健康数据；未受信浏览器需输入访问密码，验证后生成设备标识并标记为受信设备；不提供注册、邀请、多用户管理或账号资料体系。

### NonFunctional Requirements

NFR1: 健康记录、跑步记录、目标和提醒配置均属于个人敏感数据，不应被公开访问。

NFR2: MVP 采用访问密码和受信浏览器设备标识作为轻量访问保护；该方案不等同于强认证系统，后续公开部署或多人使用时必须重新评估认证方案。

NFR3: 核心记录流程必须在移动浏览器和桌面浏览器上可完成。

NFR4: 创建一次健康记录和一次跑步记录的主路径应控制在 1 分钟内。

NFR5: 编辑或删除记录后，首页趋势和目标进度必须反映最新数据。

NFR6: 预计达成时间必须能在数据不足、低于配置阈值或趋势反向时明确说明“无法估算”或等价中文提示。

NFR7: 鼓励反馈应具体、克制、支持行动，不使用羞辱性语言、医疗化判断或夸张确定性。

NFR8: SMTP 密码或授权码不应在配置界面明文回显。

NFR9: 访问密码不得明文保存；服务端仅保存访问密码哈希。

NFR10: 图表和进度不得只依赖颜色表达含义，必须有中文文字摘要或标签。

NFR11: 响应式网页应用目标为 WCAG 2.2 AA；所有输入必须有可见标签和单位，错误信息要和字段关联，并能被读屏识别。

NFR12: 移动端点击目标至少 44px 高，所有可交互控件都有可见焦点状态。

NFR13: 可见 UI 文案使用中文，产品显示名为“瘦身助手”。

### Additional Requirements

- 使用模块化单体架构，按功能切片分层：`app/`、`features/*/actions`、`features/*/services`、`features/*/repositories`、`db/`、`integrations/`。

- 所有写入从 `app/` 进入 `features/*/actions/` 或 route handler，再进入 `features/*/services/`；UI、route handler、scheduler 不得直接写数据库。

- MVP 使用一个本地 SQLite 数据库。应用代码只能通过 Drizzle repository 访问数据库；禁止在 UI、server action 和 scheduler 中散落 SQL。

- BMI、趋势、周汇总、目标差距、预计达成时间由 query/read-model service 按需计算；数据库只存源事实和配置。

- 服务器本地保存访问密码哈希和受信设备记录；所有读取和写入个人数据的 server entrypoint 都必须校验受信设备。

- 提醒条件只由 `ReminderRunner` 调用提醒服务计算；页面只展示提醒状态，不重新判定发送资格。

- `ReminderRunner` 从 SQLite 读取提醒设置和最新记录，创建站内提醒，并在邮件提醒启用且 SMTP 有效时通过 Nodemailer 发送邮件。

- 用户可配置项必须落在 typed settings model 中：个人资料、SMTP、趋势估算阈值、提醒规则、访问密码元数据、受信设备。

- 环境变量只允许用于部署级路径和非用户可编辑默认值。

- MVP 运行在一个长运行 Node.js 进程中，并拥有本地文件系统写权限；不要部署到无持久磁盘或无常驻进程的 serverless/edge 环境。

- 数据目录必须可持久化备份。

- 数据库主键使用字符串 UUID；URL 和 server actions 不暴露自增序列。

- 记录按用户本地日期归属，日期字段保存 `YYYY-MM-DD`，时间戳保存 ISO 字符串。

- 内部单位约定：体重 kg、距离 km、围度 cm、心率 bpm、步幅 m、步频 spm、配速秒/公里，界面格式化为中文单位。

- `healthRecord` 对 `localDate` 唯一；`runRecord` 对同一 `localDate` 允许多条；`goal` 按类型区分；`setting` 按 key/type 区分为单例配置；`reminderEvent` 按 `localDate + reminderType + channel` 幂等。

- 技术栈种子：Node.js 24.18.0 LTS、Next.js 16.2.9、React 19.2、TypeScript 6.0.3、Tailwind CSS 4.3.1、shadcn/ui CLI v4、Drizzle ORM 0.45.2、Drizzle Kit 0.31.10、better-sqlite3 12.11.1、Nodemailer 9.0.1、SQLite。

### UX Design Requirements

UX-DR1: 实现中文可见界面，产品显示名为“瘦身助手”；不得出现英文主界面文案，除必要技术标识如 SMTP、STARTTLS、BMI。

UX-DR2: 实现简洁、数据优先、低装饰的视觉风格，遵守 `DESIGN.md` 中的色彩、字体、圆角、间距和组件规则。

UX-DR3: 实现响应式导航：桌面端左侧常驻导航；移动端顶部标题栏和底部导航：首页、记录、历史、目标、设置。

UX-DR4: 实现访问验证页面：首次访问创建访问密码；未受信浏览器输入访问密码；受信浏览器跳过访问验证。

UX-DR5: 实现首页仪表盘：今日状态、健康摘要、运动摘要、目标进度、预计达成时间、鼓励文案、7 天 / 30 天趋势卡。

UX-DR6: 实现今日记录页面：健康记录表单和跑步记录表单分组展示；健康记录同日覆盖；跑步记录同日可多条；所有数值输入显示单位。

UX-DR7: 实现历史页面：支持按全部/健康/跑步筛选，支持周期筛选，支持查看、编辑和删除记录。

UX-DR8: 实现目标页面：支持健康目标和跑步目标分组设置，并展示当前目标进度摘要。

UX-DR9: 实现提醒页面：支持提醒时间、站内提醒、邮件提醒配置，并清楚显示 SMTP 配置依赖。

UX-DR10: 实现设置页面：个人资料、SMTP、趋势估算阈值、访问密码、受信设备分组展示，每组独立保存。

UX-DR11: 实现指标卡组件：显示名称、数值、单位、变化说明和领域颜色；点击可进入相关趋势详情或历史筛选。

UX-DR12: 实现目标进度卡组件：显示当前值、目标值、剩余差距、进度条和预计达成时间；无法估算时显示原因。

UX-DR13: 实现今日状态条组件：展示健康记录、跑步记录和提醒状态，每一项可跳转到对应操作。

UX-DR14: 实现图表卡组件：支持 7 天和 30 天视图；数据不足时显示空状态；图表旁必须有文字摘要。

UX-DR15: 实现提醒条组件：满足提醒条件时在页面内显示，不使用弹窗阻塞用户操作。

UX-DR16: 实现设置分组和受信设备行组件：显示浏览器/设备名称、创建时间、最近访问时间和移除操作。

UX-DR17: 实现表单错误体验：数值字段非法时，字段旁显示错误，阻止保存，但保留已输入内容。

UX-DR18: 实现破坏性操作确认：删除记录、移除受信设备、清空 SMTP 配置都需要确认。

UX-DR19: 避免第一版禁止交互：拖拽排序、轮播仪表盘、移动端悬停操作、两层以上弹窗、打卡游戏化动画、核心历史记录无限滚动。

UX-DR20: 实现视觉稿覆盖的页面布局参考：访问验证、首页、今日记录、历史、目标、提醒、设置，参考 `mockups/key-screens.html`。

## Epic 1: 应用基础与访问保护

用户可以首次进入瘦身助手，创建访问密码，后续通过受信浏览器访问受保护的中文应用框架。

### Story 1.1: 初始化应用骨架与中文布局

As a 个人用户,
I want 打开瘦身助手时看到中文应用框架,
So that 我可以在统一的受保护界面中使用后续功能.

**Acceptance Criteria:**

**Given** 项目尚未有应用代码
**When** 开发者初始化应用
**Then** 应用使用 Next.js、React、TypeScript、Tailwind、shadcn/ui 基础栈
**And** 页面显示名为“瘦身助手”
**And** 可见主界面文案为中文
**And** 桌面端具备左侧导航框架
**And** 移动端具备可适配的导航结构
**And** 不出现营销式首页或英文主界面

### Story 1.2: 建立本地数据库与访问保护数据模型

As a 系统,
I want 保存访问密码哈希和受信设备记录,
So that 个人健康数据可以被轻量保护.

**Acceptance Criteria:**

**Given** 应用需要保存访问保护状态
**When** 数据库 schema 初始化
**Then** 系统创建访问密码元数据和受信设备相关表
**And** 访问密码只能以哈希形式保存
**And** 受信设备保存服务端可校验的设备标识记录
**And** 数据访问通过 Drizzle repository，不在 UI 或 route handler 中散落 SQL
**And** 数据库使用本地 SQLite

### Story 1.3: 首次访问创建访问密码

As a 首次访问的个人用户,
I want 第一次打开应用时创建访问密码,
So that 后续访问个人健康数据前有基本保护.

**Acceptance Criteria:**

**Given** 服务器本地还没有访问密码
**When** 用户打开应用
**Then** 系统显示中文“创建访问密码”页面
**And** 用户可以输入并确认访问密码
**And** 密码不一致或不符合校验规则时显示中文字段错误
**And** 创建成功后服务端只保存密码哈希
**And** 当前浏览器被标记为受信设备
**And** 用户进入受保护的应用框架

### Story 1.4: 未受信浏览器访问验证与受信设备管理基础

As a 已设置访问密码的个人用户,
I want 新浏览器需要输入访问密码，而受信浏览器可以直接进入,
So that 我的个人健康数据不会被未验证浏览器直接访问.

**Acceptance Criteria:**

**Given** 服务器本地已有访问密码
**When** 未受信浏览器打开应用
**Then** 系统显示中文访问密码验证页面
**And** 输入错误密码时显示中文错误，不暴露底层错误
**And** 输入正确密码后浏览器生成设备标识并保存到 cookie 或 localStorage
**And** 服务端记录该浏览器为受信设备
**And** 受信浏览器再次访问时跳过验证
**And** 未受信浏览器不能访问受保护页面或 server entrypoint
**And** 清除 cookie/localStorage、更换浏览器或隐私模式访问时需要重新验证

## Epic 2: 每日记录与历史管理

用户可以快速记录当天健康数据和跑步数据，并在历史页面查看、编辑、删除记录。

### Story 2.1: 建立健康记录和跑步记录数据模型

As a 系统,
I want 保存健康记录和跑步记录的源事实数据,
So that 后续记录、历史和趋势都基于一致的数据.

**Acceptance Criteria:**

**Given** 用户需要保存健康和跑步记录
**When** 数据库 schema 扩展
**Then** 系统创建健康记录表，且 `localDate` 唯一
**And** 系统创建跑步记录表，同一 `localDate` 可有多条
**And** 记录使用字符串 UUID 主键
**And** 日期字段保存为 `YYYY-MM-DD`
**And** 体重、距离、围度、心率、步幅、步频、配速按架构约定保存为标准内部单位
**And** repository 提供创建、更新、查询、删除记录的基础方法

### Story 2.2: 录入和覆盖当天健康记录

As a 个人用户,
I want 录入或修改当天健康数据,
So that 每天只保留一条有效身体记录.

**Acceptance Criteria:**

**Given** 用户在受信浏览器进入“今日记录”页面
**When** 用户填写体重、腰围、腿围、体脂率并保存
**Then** 系统创建当天健康记录
**And** 如果当天已有健康记录，系统更新原记录而不是新增第二条
**And** 表单显示中文标签和单位
**And** 非法数值在字段旁显示中文错误
**And** 保存成功后显示中文成功提示
**And** 该记录可被后续首页和历史页面读取

### Story 2.3: 添加当天跑步记录

As a 个人用户,
I want 录入一次跑步活动,
So that 一天内多次跑步都能被单独记录.

**Acceptance Criteria:**

**Given** 用户在“今日记录”页面
**When** 用户填写公里数、运动时长、平均心率、平均步幅、步频并保存跑步记录
**Then** 系统创建一条跑步记录
**And** 同一天可以创建多条跑步记录
**And** 系统可以根据公里数和运动时长计算配速
**And** 用户可以手动覆盖配速
**And** 非法数值在字段旁显示中文错误
**And** 保存后记录仍归属到用户本地日期

### Story 2.4: 浏览和筛选历史记录

As a 个人用户,
I want 查看和筛选历史健康与跑步记录,
So that 我可以回顾自己的记录而不是翻表格.

**Acceptance Criteria:**

**Given** 系统中已有健康记录和跑步记录
**When** 用户打开“历史”页面
**Then** 系统按日期展示健康记录和跑步记录
**And** 用户可以按全部、健康、跑步筛选
**And** 用户可以按最近 7 天、最近 30 天或自定义日期范围筛选
